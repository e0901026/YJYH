# 手机借机管理 App — Android 测试计划

> 目的：让用户能在 Android 版本开发过程中清楚知道“怎么测、测什么、什么时候算通过”。

## 1. 测试方式

Android 版本采用四层测试：

| 层级 | 测试对象 | 谁来测 | 目的 |
|------|----------|--------|------|
| 原型对照测试 | App 页面 vs Pencil 原型 | Agent + 用户 | 确认界面、文案、按钮、导航还原原型 |
| 业务流程测试 | 扫码借、注册手机、一键还、邀请码 | Agent + 用户 | 确认流程符合 PRD |
| Android 能力测试 | 相机扫码、权限、真机表现 | Agent + 用户 | 确认 Android 设备上可用 |
| 回归测试 | 已完成页面和流程 | Agent | 确认新修改没有破坏旧功能 |

## 2. 用户怎么测试

用户测试不需要读代码，只需要按测试包和清单操作。

每次交付测试版本时，Agent 需要提供：

- Android 安装包或运行方式。
- 本次版本改动范围。
- 本次需要重点测试的页面。
- 测试账号或 mock 数据说明。
- 测试清单。
- 已知问题。

当前本地验证命令：

```sh
cd android-app
export JAVA_HOME="/Applications/Android Studio.app/Contents/jbr/Contents/Home"
export ANDROID_HOME="$HOME/Library/Android/sdk"
./gradlew :app:assembleDebug
./gradlew :app:testDebugUnitTest
./gradlew :app:lintDebug
```

当前 Debug APK：

```txt
android-app/app/build/outputs/apk/debug/app-debug.apk
```

用户测试步骤：

1. 安装或运行当前 Android 测试版本。
2. 按测试清单逐项操作。
3. 对照 `prototype/pencil-prototype.html` 看页面是否一致。
4. 发现问题时说明页面、操作步骤、期望结果和实际结果。
5. Agent 根据问题判断返工位置：PRD、原型、代码或测试用例。

## 3. Android 真机测试重点

必须优先真机验证：

- 首次启动是否正常。
- 登录 / 注册表单输入是否正常。
- 底部导航切换是否正常。
- 首页布局是否适配常见 Android 屏幕。
- 点击「扫码借」是否能申请相机权限。
- 相机权限拒绝后是否有提示。
- 相机权限允许后是否能进入扫码界面。
- 扫码失败或无法解析 IMEI 是否有兜底提示。
- 手动输入 IMEI 兜底入口是否可用。
- 长 IMEI 文本是否显示完整或可读。
- 返回键行为是否合理。

## 4. V0.2 首轮测试清单

V0.2 目标：Android 项目脚手架、页面导航、基础页面壳、全局样式、底部导航。

通过标准：

- App 能在 Android 模拟器或真机启动。
- 页面尺寸、背景、卡片、按钮、Tab 风格与原型一致。
- 首页、设备、我的底部导航可切换。
- 流程页不强行显示底部导航。
- 页面标题、主按钮、次按钮文案与原型一致。
- 没有明显文字重叠、裁剪或不可读。

V0.2 暂不要求：

- 真实登录接口。
- 真实扫码解析。
- 真实通知推送。
- 真实后端数据。

## 5. V0.3 核心流程测试清单

V0.3 目标：核心业务流程可用，可先使用 mock 数据。

必须测试：

- 登录后进入首页。
- 注册页字段完整：邀请码、工号、名称、密码、确认密码。
- 首页展示手上持有台数。
- 点击「扫码借」进入扫码流程。
- 已建档 IMEI 展示设备信息并可确认借走。
- 未建档 IMEI 进入手机注册页。
- 手机注册页填写手机名称后可完成建档并借走。
- 一键还列表展示待归还设备。
- 点击单条「一键还」后状态变化。
- 设备列表筛选可切换。
- 设备详情展示 IMEI、当前持有人、绑定 owner、当前状态。
- 我的页展示工号、名称、邀请码用量。
- Owner 后台展示用户列表和邀请码列表。

## 6. 问题记录格式

发现问题时按这个格式记录：

```txt
页面：
操作步骤：
期望结果：
实际结果：
截图/录屏：
是否阻塞继续测试：
```

问题分类：

- PRD 问题：需求规则不清或错误。
- 原型问题：原型没有表达或表达不一致。
- 开发问题：实现没有还原原型或功能不可用。
- 测试问题：清单缺漏或测试数据不足。

## 7. 通过标准

一个版本可以进入下一阶段，需要满足：

- 本版本范围内的阻塞问题为 0。
- 高优先级问题均已修复或明确延后。
- 页面实现与原型主要结构一致。
- 核心业务流程符合 PRD。
- Android 真机关键能力验证通过。

## 8. 测试交付物

每轮测试结束后应形成：

- 测试版本号。
- 测试范围。
- 通过项。
- 问题列表。
- 延后项。
- 是否允许进入下一阶段。
