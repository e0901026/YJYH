# V0.4 范围确认

> 阶段目的：在进入下一轮 Android 开发前，先确认真实 Android 能力、原型补充点、后端接口契约和测试门禁，避免 V0.4 边开发边改方向。

## 1. 版本定位

V0.4 建议定位为：Android 真实扫码能力与后端联调准备版。

V0.4 不直接切换真实后端，不实现 Spring Boot 服务端；后端在本阶段只细化 API 契约、数据库字段和业务事务，为后续 V0.5 后端 MVP 做准备。

## 2. V0.4 目标

- Android 扫码借从纯手动 mock 过渡到真实相机扫码入口。
- 支持相机权限申请、权限拒绝提示、扫码失败提示和手动输入兜底。
- 解析条码/二维码中的 IMEI，继续复用 V0.3 已验证的已建档 / 未建档流程。
- 明确通知类型和触发条件：借走通知、催还通知、归还通知。
- 细化后端 API 契约和数据库字段，不在本阶段实现服务端。
- 补充 V0.4 测试清单和截图级视觉验收要求。

## 3. 范围内

### Android

- 新增相机权限声明。
- 新增扫码运行态：等待权限、权限被拒绝、扫码中、解析成功、解析失败。
- 接入 CameraX + ML Kit Barcode Scanning。
- 解析结果必须提取 15 位 IMEI；无法提取时显示失败提示。
- 保留手动输入 15 位 IMEI 的兜底入口。
- 已建档 IMEI 继续进入确认借走流程。
- 未建档 IMEI 继续进入注册手机流程。
- 不把完整 IMEI 写入普通调试日志。

### 原型

V0.4 开发前必须先补充或确认以下原型状态：

- 相机权限申请说明态。
- 权限拒绝后的提示态。
- 扫码失败 / 无法解析 IMEI 的提示态。
- 手动输入兜底入口。
- 扫码成功后的已建档 / 未建档分支仍保持当前结构。

### 通知契约

- 借走通知：借机成功后通知上一位持有人和设备绑定 owner。
- 催还通知：借出方点击「催还机」后通知当前持有人。
- 归还通知：借入方点击「一键还」后通知设备 owner 和上一位相关方。
- V0.4 只做通知记录/模拟反馈契约，不做真实系统推送。

### 后端契约

- 补充 API 请求/响应字段。
- 补充错误码：无权限、IMEI 无效、设备不存在、设备已被当前用户持有、邀请码无效等。
- 补充通知类型枚举。
- 确认数据库字段是否支撑 Android 当前页面。

## 4. 范围外

- 不实现真实后端服务。
- 不接真实登录接口。
- 不接真实推送服务。
- 不做后台 Web 管理端。
- 不做正式发布、签名分发和生产环境部署。
- 不做 iOS / Web 版本。

## 5. 开发准入

进入 V0.4 Android 开发前必须满足：

- PRD 已补充扫码权限、扫码失败和手动兜底规则。
- Pencil 原型已补充 V0.4 新增状态并导出预览图。
- 用户确认 V0.4 原型可进入开发。
- Android 技术架构已明确 CameraX / ML Kit 依赖和测试方式。
- 测试清单已写清楚真机 / 模拟器分别怎么测。

## 6. 验收标准

- App 能在模拟器或真机启动。
- 点击「扫码借」后能进入扫码页。
- 首次进入扫码页能触发相机权限流程。
- 权限拒绝后有明确提示和手动输入兜底。
- 权限允许后可进入扫码中状态。
- 扫码失败或解析不到 IMEI 时有提示。
- 手动输入兜底仍能跑通已建档 / 未建档两条路径。
- 页面截图对照 Pencil 原型无明显文字截断、重叠、遮挡。
- `assembleDebug`、`lintDebug`、`testDebugUnitTest` 通过。

## 7. 建议开发顺序

1. 更新 PRD、Pencil 原型和测试清单。
2. 增加 Android 相机权限和依赖。
3. 新增扫码 UI 状态和权限拒绝提示。
4. 接入 CameraX 预览。
5. 接入 ML Kit Barcode Scanning 并解析 IMEI。
6. 保留手动输入兜底并复用 V0.3 流程。
7. 做模拟器截图检查、真机权限检查和功能回归。

## 8. 待用户确认

- V0.4 是否按“真实 Android 扫码能力优先”推进。
- V0.4 是否暂不实现真实后端，只做后端契约准备。
- V0.4 是否需要把“通知中心/通知列表”做成页面，还是继续以状态反馈和后端通知记录为主。
