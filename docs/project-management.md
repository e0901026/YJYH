# 手机借机管理 App — 项目管理

> 目的：作为本项目的轻量任务控制台，管理需求、原型、开发、测试四个环节的状态，确保每次迭代都有明确入口、范围和验收口径。

## 1. 项目定位

- 项目类型：企业内部移动端工具。
- 平台优先级：Android App 优先开发。
- GitHub 仓库：`https://github.com/e0901026/YJYH`（Public）
- 当前阶段：V0.8 Owner 用户管理真实 CRUD 与邀请码生成联调已完成本地验证；下一步整理最新 APK 验收包并继续补齐剩余真实业务闭环。
- 工作方式：单 Agent 负责维护 PRD、原型、开发与测试闭环。
- 跨电脑接手：通过 GitHub clone 仓库后，在 Codex 中添加项目文件夹，并先阅读 `Agent.md` 与 `docs/codex-handoff.md`。
- 最高约束：需求先行，原型确认后再开发，开发必须还原原型。
- 最终交付：提供可安装 Android APK 给用户人工使用，且具备行为埋点、错误日志和问题诊断能力。

## 2. 当前版本目标

### V0.5 后端 MVP 范围确认版

目标：在 Android 真实扫码和核心 mock 流程通过后，开始建设真实后端的最小可用版本，为后续 Android 从 mock repository 切换到真实 API 做准备。

范围：

- Spring Boot 后端工程脚手架。
- PostgreSQL 数据模型和 Flyway 迁移。
- 登录 / 注册 / token 认证。
- 设备查询、IMEI 查询、设备建档。
- 借机、归还、催还通知记录。
- 邀请码申请、Owner 邀请码生成、Owner 用户列表。
- 后端单元测试 / 接口测试 / CI 检查。
- API 返回结构对齐 Android 当前页面字段。
- App 行为事件和错误日志接收接口。

不在 V0.5 范围：

- Android 全量切换真实 API。
- 生产部署和正式域名。
- 真实推送服务。
- 后台 Web 管理端。
- 复杂审批流、审计导出和多租户。

范围文档：`docs/v0.5-scope.md`

### V0.4 Android 真实扫码能力与后端联调准备版

目标：在 V0.3 mock 闭环通过的基础上，进入真实 Android 能力准备，优先补齐相机权限、扫码失败、手动输入兜底和通知契约。

范围：

- Android 相机权限申请、拒绝提示和重新授权路径。
- CameraX + ML Kit Barcode Scanning 技术方案确认。
- 扫码失败 / 无法解析 IMEI 的兜底流程。
- 手动输入 IMEI 兜底继续可用。
- 已建档 / 未建档两条流程继续复用 V0.3 状态闭环。
- 借走通知、催还通知、归还通知的触发规则与后端契约。
- 后端 API 契约和数据库字段细化，不实现真实服务端。

不在 V0.4 范围：

- 真实后端服务实现和部署。
- 真实推送服务。
- 后台 Web 管理端。
- iOS、Web 或多端开发。

范围文档：`docs/v0.4-scope.md`

### V0.1 原型确认版

目标：确认核心业务流程、页面结构和统一视觉风格。

范围：

- 登录 / 注册。
- 首页：手上持有台数、扫码借、一键还、邀请码状态。
- 扫码借：借机人扫码、识别已建档手机、确认借走、通知上一位持有人和设备 owner。
- 手机注册：未建档 IMEI 填写手机名称并建档。
- 一键还：当前待归还列表和单项归还。
- 设备：设备列表、筛选、设备详情、owner 信息。
- 我的：个人信息、邀请码申请、Owner 后台入口。
- Owner 后台：用户列表、邀请码管理。

不在当前版本范围：

- 复杂多 Agent 工作流。
- 后台管理系统独立 Web 端。
- iOS、Web 或多端同步开发。
- 真实后端服务实现和部署。
- 精细权限矩阵之外的高级审批流。
- IMEI 合规导出、审计报表等扩展能力。

## 5. 开发任务拆分

进入开发前必须先完成当前页面的原型确认。

建议开发顺序：

1. Android 项目脚手架与全局样式。
2. Android 页面导航、页面壳、底部导航。
3. 登录与注册。
4. 首页。
5. 扫码借与手机注册流程，优先验证 Android 相机扫码能力。
6. 一键还。
7. 设备列表与设备详情。
8. 我的与邀请码申请。
9. Owner 用户列表与邀请码管理。
10. 状态流、通知模拟、Android 真机测试验收。

每个开发任务必须包含：

- 对应 PRD 章节。
- 对应原型页面。
- 主要交互。
- 完成标准。
- 测试检查点。

## 6. 原型确认规则

页面进入开发前必须满足：

- 页面业务逻辑与 `docs/PRD-手机借机管理.md` 一致。
- 页面视觉与 `docs/system-design-system.md` 一致。
- 页面已在 `prototype/手机借机管理.pen` 中保存。
- 页面已导出到 `prototype/exports` 并能在 `prototype/pencil-prototype.html` 中查看。
- 用户明确表示该页面可进入开发。

如果用户要求调整界面：

1. 先判断是否属于业务需求变更。
2. 若是业务变更，先更新 PRD。
3. 再更新 Pencil 原型。
4. 用户确认原型后，必要时再沉淀设计规范。
5. 最后进入开发或修改代码。

## 7. 测试验收规则

每个功能完成后按三层验收：

| 层级 | 验收对象 | 重点 |
|------|----------|------|
| PRD 验收 | 业务规则 | 角色、权限、流程、异常是否正确 |
| 原型验收 | 界面与交互 | 页面结构、文案、按钮、Tab、状态是否还原 |
| 功能验收 | 实现结果 | 跳转、提交、状态变化、错误提示是否可用 |

测试结果必须记录：

- 通过项。
- 偏差项。
- 偏差来源：PRD / 原型 / 开发 / 测试。
- 需要返工的页面或文件。
