# 手机借机管理 App — 产品需求文档（原型用）

> 用途：交给 Pencil / 其他原型工具或设计助手，按本文搭建界面与交互流程。
> 平台假设：Android App 优先开发，移动端竖屏手机，企业内部使用。

## 1. 背景与目标

用户持有大量手机，并会与同事互相借机体验。经常出现：记不清手机现在被谁拿走、手上这台该还给谁、当前有哪些机可以借。

目标：用最少操作完成「登记 → 扫码借走 → 归还」，并在首页一眼看到手上持有台数。

当前开发优先级：先开发 Android 版本。原型、开发计划、技术选型和测试验收均以 Android App 为优先目标；其他平台如需支持，必须另行补充需求和计划。

## 2. 角色

| 角色 | 说明 |
|------|------|
| 普通用户 | 注册后可扫码借、一键还、管理与自己相关的设备与借用记录；可申请邀请码（有配额）。 |
| 项目 Owner | 无限生成邀请码；可查看全员列表及邀请关系；其余能力与业务规则同普通用户，除非单独扩展后台权限。 |

## 3. 账号与注册

### 3.1 注册资格

- 必须持有有效邀请码才可注册；无邀请码不可完成注册。

### 3.2 注册必填字段

1. 邀请码
2. 工号（登录标识之一，全局唯一，同一工号只能注册一次）
3. 名称（展示名）
4. 密码（含确认密码；具体强度策略由实现决定）

### 3.3 登录

- 使用工号 + 密码登录。

### 3.4 邀请码规则

- Owner 可无限次生成邀请码。
- 普通用户可自行申请邀请码；每名用户终身累计最多 10 张，已使用、已过期、已作废均计入配额。
- 达到 10 张后禁止再申请，提示联系 Owner 或说明配额已满。
- 每名注册用户需记录邀请人。Owner 在用户列表中必须能看到：工号、名称、注册时间、邀请人（名称 + 工号）。

## 4. 核心业务概念

### 4.1 设备唯一标识

- 设备以完整 IMEI（及双卡场景下的 IMEI2，若有）作为权威硬件标识，完整入库，用于避免纠纷。
- 录入方式：扫描手机工程界面中展示的条码/二维码（如拨号盘常见 `*#06#` 等出现的 IMEI 码）；支持手动输入 IMEI 作为兜底。
- 不使用机身外贴的固定二维码作为唯一绑定方式；贴纸可作为辅助，但系统主键为 IMEI。

### 4.2 设备名称与 owner

- 首次扫码识别到未建档 IMEI 时，进入手机注册页，由当前扫码用户填写手机名称并完成建档。
- 系统需要记录设备绑定 owner。owner 是设备档案的责任人，用于接收借机通知和追溯记录。
- 仅设备绑定 owner 可修改设备名称和设备档案信息。
- owner 可将设备转让给其他用户；转让结束后仅新 owner 可改名。
- 任意时刻每台设备只有一个绑定 owner。

### 4.3 当前持有人与借用记录

- 扫码借由借机人发起：借机人点击首页「扫码借」，调起相机扫描要借走的手机 IMEI。
- 若该 IMEI 已建档，系统展示被借手机的详细信息；借机人确认后即借走该手机。
- 系统自动记录当前借机人为最新持有人，不需要手动选择对方。
- 系统自动结束或更新上一段持有关系，并生成新的未结束借用记录。
- 系统向上一位持有人和设备绑定 owner 发送「手机已被某某借走」的通知，用于记录和追溯。
- 首页「手上持有：N 台」由规则自动计算，无需手改数字。

## 5. 核心功能（必须实现于原型）

### 5.1 扫码借

入口：首页主按钮「扫码借」。

流程：

1. 借机人点击「扫码借」后调起相机扫码。
2. 系统解析 IMEI 或等价条码内容。
3. 若 IMEI 未建档：进入手机注册页，展示完整 IMEI，用户填写手机名称并确认建档。
4. 若 IMEI 已建档：展示被借手机的详细信息，包括设备名、完整 IMEI、当前持有人、绑定 owner、当前状态。
5. 借机人点击「确认借走」。
6. 系统自动记录借机人为最新持有人，并通知上一位持有人和设备绑定 owner。

原型需体现：点击扫码借 → 调起相机扫码 → 相机权限说明 / 权限拒绝 / 扫码失败 / 手动输入 IMEI 兜底 → 已建档手机确认借走 → 成功反馈；同时体现未建档手机注册页。

### 5.2 一键还

入口：首页主按钮「一键还」。

- 列出当前所有未结束的借用，区分我借出去的 / 我借入的。
- 我借入的记录展示设备名称、对方和「一键还」按钮，可选二次确认。
- 我借出去的记录展示设备名称、当前持有人、持有人工号、IMEI、持有天数，并提供「催还机」按钮；点击后给当前持有人发送催还消息，不改变设备归还状态。
- 我借出去的记录不提供「一键还」按钮；归还动作应由当前持有人在「我借入的」中发起。
- 列表项右侧不得显示「我借出去的」「我借入的」这类身份标签；借出 / 借入的区分由顶部 Tab 承担。
- 原型需体现：借出 Tab 为设备持有人信息列表 + 右侧「催还机」；借入 Tab 为可归还列表 + 右侧「一键还」。

### 5.3 手上持有台数

- 首页顶部显著展示：例如「手上持有：12 台」。
- 可点击进入「手上设备」列表。
- 与「扫码借」「一键还」同级可见。

## 6. 信息架构与页面清单

底部 Tab（三栏）：首页 · 设备 · 我的

Owner 专属：从「我的」进入管理后台。

通用导航：

- 流程页返回入口放在标题栏左侧，使用系统常规返回样式。
- 标题栏右侧只保留当前页主要补充动作；设备列表添加设备入口显示为 `+`。
- 设备相关入口使用明确的手机/设备图标，避免抽象或难以识别的符号。

| 序号 | 页面 | 要点 |
|------|------|------|
| 1 | 登录 | 工号、密码；跳转注册 |
| 2 | 注册 | 邀请码、工号、名称、密码、确认密码 |
| 3 | 首页 | 手上持有 N 台；扫码借；一键还；邀请码申请入口（显示已用 x/10） |
| 4 | 扫码借 - 识别与确认 | 调起相机扫码；已建档手机展示详情；确认借走；通知上一位持有人和绑定 owner |
| 5 | 手机注册 | 未建档 IMEI 展示完整 IMEI；填写手机名称；确认建档 |
| 6 | 一键还 | 我借入的可一键还；我借出去的展示设备名、当前持有人、工号、IMEI、持有天数，并可催还机 |
| 7 | 设备列表 | 筛选：全部 / 在我手上 / 已借出 / 借入待还；添加设备（扫 IMEI） |
| 8 | 设备详情 | 名称、完整 IMEI、当前持有人、绑定 owner、当前状态、转让 owner、流水入口 |
| 9 | 我的 | 名称、工号；申请邀请码（x/10）；修改密码；退出；跳转 Owner 后台（若已是 Owner） |
| 10 | Owner - 用户列表 | 搜索；每行：工号、名称、注册时间、邀请人 |
| 11 | Owner - 邀请码 | 生成新码（无上限）；列表状态（未使用/已使用/已过期等） |

## 8. 异常与边界

- 邀请码无效 / 过期
- 工号已被注册
- IMEI 已被他人绑定但信息可追溯
- 扫码失败或无法解析 IMEI
- 首次使用扫码借时需要申请相机权限
- 相机权限被拒绝时，需要展示提示并保留手动输入 IMEI 兜底
- 权限允许但无法识别条码 / 二维码时，需要提示重新扫描或手动输入
- 未建档 IMEI 需要先注册手机名称
- 非 owner 点击编辑设备名称 → 无权限或隐藏入口
- 用户邀请码配额已满

## 9. 非功能需求

- 密码存储哈希；传输 HTTPS。
- IMEI 敏感：最小权限展示，审计可查完整 IMEI（owner/合规导出）。
- 最终交付必须提供可安装 Android APK，供用户人工使用和验收。
- App 必须具备基础行为埋点、错误日志和问题反馈收集能力，用于定位用户遇到的 bug 和后续优化。
- 埋点和日志必须覆盖关键路径：登录/注册、扫码借、权限拒绝、扫码失败、手动输入、确认借走、手机注册、一键还、催还机、设备详情、邀请码、Owner 页面。
- 发生异常、接口失败、权限失败、扫码无法解析、页面状态不一致时，必须记录可诊断日志。
- 日志和埋点不得泄露完整密码、token；IMEI 默认脱敏，只有合规授权场景才能查看完整值。

## 10. 原型设计一致性要求

- 所有页面使用统一移动端低保真视觉体系：浅灰页面背景、白色卡片、蓝色主按钮、浅色描边、8px 左右圆角。
- 首页、扫码借、一键还、设备、我的、Owner 后台的标题、卡片、按钮、Tab 样式需保持一致。
- 首页作为核心入口，可以突出「手上持有」数字，但不得使用与其他页面明显割裂的黑色大块按钮或不同风格控件。
- 主操作使用蓝色实心按钮；次操作使用白底蓝色描边或浅色卡片。

## 11. 开发实现约束

- 后续真实开发必须严格按照已确认原型进行。
- 原型是页面结构、主要文案、按钮层级、状态展示和核心交互的实现基准。
- 若开发中需要改变界面或流程，必须先更新本 PRD 和原型，再进入代码实现。
- 验收时必须同时检查：业务是否符合本 PRD、界面与交互是否符合原型。

## 12. 需求、原型、开发、测试闭环

- 需求文档：定义业务规则、角色、流程、页面清单、关键文案和边界。
- 原型：跟随需求文档，表达页面、流程、状态和统一视觉风格。
- 前端开发：完全还原原型，不自行改变页面结构、文案、样式和交互。
- 测试验收：同时对照需求文档和原型，检查业务正确性、原型还原度和风格一致性。
- 返工规则：需求不清先改需求；原型缺失先补原型；开发偏离原型则改开发；测试缺少覆盖则补测试。
