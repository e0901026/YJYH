# 手机借机管理 App — 需求 / 原型 / 开发 / 测试闭环

> 目的：确保后续每次迭代都能从需求稳定落到原型，再从原型稳定落到前端实现，最后被测试验证。

## 1. 闭环总原则

```txt
需求文档 → 原型 → 前端开发 → 测试验收 → 返工修正
```

- 需求文档是业务真相。
- 原型是界面和交互真相。
- 前端开发必须完全还原原型。
- 测试必须同时检查需求、原型和实现是否一致。

## 2. 需求文档

需求文档负责说明：

- 用户角色和权限。
- 核心业务流程。
- 页面清单和每页要点。
- 关键文案。
- 异常、边界和非功能要求。
- 开发和验收约束。

如果用户提出新的业务规则，先更新 PRD，再更新原型。

## 3. 原型

原型负责承接需求文档：

- 每个核心流程必须在原型中可见。
- 每个页面必须表达主要信息、按钮、状态和反馈。
- 页面风格必须遵守 `docs/system-design-system.md`。
- 原型入口是 `prototype/pencil-prototype.html`。
- 页面导出图位于 `prototype/exports/*.png`。

如果 PRD 有需求但原型没表达，不能直接进入前端开发，必须先补原型。

## 4. 前端开发

前端开发负责完全还原原型：

- 页面结构按原型实现。
- 文案按原型实现。
- 主次按钮、卡片、列表、Tab、状态标签按原型实现。
- 页面跳转和核心交互按原型实现。
- 视觉风格按 `docs/system-design-system.md` 实现。

开发中不得自行改变原型。若发现原型不合理，先返回 PRD 和原型修正。

## 5. 测试验收

测试按三层检查：

1. PRD 验收：业务规则、角色权限、流程和边界是否符合需求文档。
2. 原型验收：页面结构、文案、样式、状态和交互是否还原原型。
3. 功能验收：实现是否可运行，入口、跳转、提交、状态更新是否有效。

测试结果必须标明：

- 通过项。
- 偏差项。
- 偏差属于需求、原型、开发还是测试覆盖问题。
- 需要返工的文件或页面。

## 6. 返工路径

| 问题 | 返工位置 |
|------|----------|
| 业务规则错或缺 | 先改 PRD |
| 页面没有表达需求 | 先改原型 |
| 风格不一致 | 先查设计规范，再改原型或前端 |
| 前端与原型不一致 | 改前端 |
| 测试漏验 | 补测试清单或测试用例 |

## 7. 当前项目硬约束

- 原型必须明确跟随需求。
- 风格必须保持一致。
- 前端开发必须完全还原原型。
- 测试必须把 PRD、原型、实现放在同一张验收闭环里检查。
