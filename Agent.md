# Agent Principles

本项目采用单 Agent、原型优先的工作方式：一个 Agent 直接围绕 PRD、设计规范和原型推进，减少中间层，避免需求在转述中折损。

## 必读文件

每次修改前必须先阅读或确认以下文件的相关章节：

1. `docs/PRD-手机借机管理.md`：原始需求源头，业务规则以此为准。
2. `docs/system-design-system.md`：视觉和交互一致性规范。
3. `docs/delivery-loop.md`：需求、原型、开发、测试闭环。
4. `docs/project-management.md`：当前版本范围、页面状态、开发准入和验收管理。
5. `docs/android-technical-architecture.md`：Android 技术选型、工程结构和开发边界。
6. `docs/backend-database-architecture.md`：后端框架、数据库模型和 API 边界。
7. `docs/cicd-rules.md`：CI/CD 质量门禁、测试包交付和分阶段流水线规则。
8. `docs/github-version-control.md`：GitHub 版本管理、提交范围、分支和回滚规则。
9. `docs/android-test-plan.md`：Android 测试方式、测试清单和版本通过标准。
10. `prototype/pencil-prototype.html` 与 `prototype/exports/*.png`：原型评审入口和后续开发的界面实现基准。

如果用户口头提出新规则，先更新对应文档，再改原型或代码。

## 需求优先

- 所有产品、界面、流程、文案和原型修改都必须围绕需求文档展开。
- 修改任何界面需求前，必须先更新对应的 PRD，确保需求文档能解释界面变化。
- 如果用户口头纠正了业务规则，先把纠正写入 PRD，再修改原型或代码。
- 原型只表达 PRD 中已经明确的流程，不自行添加未被需求支撑的交互。
- 任何实现、测试、评审都必须回到原始 PRD，不依赖聊天里的二手总结。

## 需求到交付闭环

- 需求文档是业务源头；原型必须明确跟随需求文档。
- 原型是前端实现源头；前端开发必须完全还原原型。
- 测试是闭环检查；测试必须同时检查需求、原型和开发实现三者一致。
- 任一环节发现不一致，必须回到上游修正：需求错先改 PRD，原型缺先补原型，实现偏差则改代码，测试缺口则补测试。

## 阶段门禁

- 每完成一个阶段或一轮任务后，必须明确给出：完成项、验证结果、是否通过、阻塞项、下一步动作。
- 阶段未通过时，不能假装进入下一阶段；必须先推进阻塞项，或明确记录阻塞原因和解决路径。
- 阶段通过后，必须更新项目管理文档、提交完整变更、推送 GitHub，再进入下一阶段。
- 如果用户没有要求暂停，完成当前阶段汇报后应主动推进到下一项最高优先级任务。
- 每次最终回复都必须回答“现在处于哪一步、下一步做什么”。

## 原型一致性

- 移动端原型优先覆盖核心任务路径，再补充管理、异常和边界状态。
- 页面上的角色、按钮、状态和通知对象必须与 PRD 中的业务规则一致。
- 扫码借流程由借机人发起：借机人扫码识别手机，确认借走；系统自动记录借机人，并通知上一位持有人和设备绑定 owner。
- 未建档手机扫码后进入手机注册流程，由借机人填写手机名称并完成建档。
- 页面风格必须遵守 `docs/system-design-system.md`，不得单页发明新风格。

## 规范变更必须先过原型

- 任何设计规范、交互规范、组件规范的新增或修改，必须先在原型中生效。
- 用户在原型中确认该规范表现后，才能把它沉淀进 `docs/system-design-system.md` 或其他长期规范文件。
- 不允许先修改规范文件，再要求原型被动跟随未经确认的新规范。
- 如果规范调整未在原型中体现，不能视为已完成。

## Pencil 源文件必须落盘

- 生成或修改 Pencil 原型时，必须确保 `prototype` 目录下存在可继续编辑的 `.pen` 源文件。
- 不能只交付 PNG、SVG、HTML 预览图；这些只能作为预览产物，不能替代 Pencil 源文件。
- 每次交付原型前，必须检查并说明 `.pen` 文件的实际路径。
- 每次通过 Pencil 修改原型后，必须显式保存，并用 `git status` 确认 `.pen` 文件出现在版本差异中；只导出 PNG 不算原型源文件已更新。
- 如果工具无法保存 `.pen` 文件，必须立即说明限制，不能让用户误以为源文件已保存。

## 开发必须遵守原型

- 任何代码改动前，必须先检查该改动是否影响界面、交互、文案或业务流程；如影响，必须先更新 PRD、Pencil 原型和预览导出，并取得用户确认后才能进入开发。
- 不允许以“实现很小”或“只是修正”为理由绕过原型确认；只要会改变用户看到或操作到的内容，就必须先走原型确认。
- 后续真实开发必须严格按照已确认原型实现，包括页面结构、核心流程、主要文案、按钮层级、状态展示和视觉风格。
- 不允许开发时绕开原型自行调整交互、增删页面、改变按钮含义或重排核心信息。
- 如果开发中发现原型不可实现、不合理或需要补充，必须先更新 PRD 和原型，确认后再写代码。
- 开发验收必须包含“实现是否匹配原型”的检查项。
- 前端像素和交互还原优先级高于开发者个人判断；若想调整，先改 PRD 和原型。

## 界面自测必须截图对照

- 任何影响界面的 Android 改动，不能只用 UIAutomator 文本结果或状态变化判断通过；必须截取模拟器实际截图，并与 `prototype/exports/*.png` 对照。
- 按钮、Tab、列表项、标题栏等核心 UI 必须检查：文字单行/换行是否符合原型、是否完整显示、是否被裁剪、是否与容器尺寸匹配。
- 对宽度较小的按钮，必须优先按原型尺寸和设计规范实现；如果真实实现会换行或截断，不能判定为通过，必须返工。
- 测试清单中所有“页面还原原型”的结论必须来自截图级检查，不能仅来自控件树文本存在。

## Changelog 必须同步

- 所有改动都必须写入 `docs/CHANGELOG.md`，包括需求变更、原型变更、设计规范变更、架构变更、代码变更、测试变更和流程纠偏。
- Changelog 必须说明：日期、变更类型、变更内容、影响范围、是否已进入开发、是否已完成用户确认。
- 如果某次改动是为了纠正流程问题，必须在 changelog 中明确记录教训和后续约束。
- 没有 changelog 的改动不能视为完整交付。

## 变更纪律

- 不修改与当前需求无关的文件。
- 不删除已有产物，除非用户明确要求或该产物已被新产物替代且说明清楚。
- 每次原型变更后，导出或更新可预览产物，并检查页面是否存在明显布局问题。
- 每次交付说明必须包含：依据文档、改动文件、验收结果、未完成事项（如有）。
- 每次完成一轮修改后，必须检查是否需要 Git 提交；提交必须包含本轮相关的需求、原型、架构、代码、测试和 CI/CD 变更。
- 不允许只提交代码而漏掉对应文档、原型或测试变更。

## 当前项目的合适架构

- 目标：内部工具原型，优先保证需求不折损、界面一致、迭代快速。
- 架构：单 Agent 直接维护 PRD、设计规范、Pencil 原型、后续前端实现和测试闭环。
- 不采用多 Agent 拆分：当前阶段不拆设计、测试、前端、后端、数据或 API Agent；除非未来系统复杂度明显上升，再重新评估。
- 原型优先：任何流程调整最终都要回到 `prototype/pencil-prototype.html` 和 Pencil 导出图里可见，而不是只停留在文档。
